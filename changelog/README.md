# Changelog

`changelog/` 记录仓库级变更、交接信息和当前工作状态。

本目录不是单文件 changelog。它采用渐进式披露：先暴露当前是否有人在工作，再暴露最近变更索引，最后才进入单次工作的详细记录。

## 先看哪里

1. 先看 [status.md](status.md)，确认当前 `工作状态`。
2. 再看 [entries/README.md](entries/README.md)，了解最近完成或正在执行的记录。
3. 需要细节时，再打开 `entries/` 下对应日期的记录文件。
4. 需要新建记录时，从 [templates/entry.md](templates/entry.md) 复制结构。

## 单 agent 工作锁

[status.md](status.md) 是本仓库的协作锁文件。任何 agent 开始工作前都必须先读取它。

- 只有 `工作状态` 为 `空闲` 时，新 agent 才能开始工作。
- 开始工作时，先把 `工作状态` 改为 `正在执行`，并填写当前 agent、工作范围、开始时间和关联记录。
- `正在执行`、`验证中`、`等待用户` 都表示仓库已被当前 agent 占用，其它 agent 不应并行修改。
- 结束工作前，把关联 entry 更新为最终状态，再把 [status.md](status.md) 改回 `空闲`。
- 如果用户明确要求接管或终止上一位 agent 的工作，新的 agent 必须先在 [status.md](status.md) 和相关 entry 中写明接管原因。

## 状态值

`status.md` 的 `工作状态` 只使用下面这些值：

- `空闲`：当前没有 agent 持有工作锁。
- `正在执行`：已有 agent 正在修改、分析或实现。
- `验证中`：已有 agent 正在跑检查或核对结果。
- `等待用户`：已有 agent 暂停等待用户确认，锁仍然有效。

单条 entry 可以额外使用 `已完成`、`已中止`、`已接管` 表达历史结果，但这些不是当前工作锁状态。

## 记录原则

- 入口短：`README.md` 只放规则和导航，不堆历史细节。
- 状态短：`status.md` 只放当前锁、范围和交接备注。
- 细节下沉：具体改动、验证和后续事项写进 `entries/`。
- 最新优先：索引和状态都把最新信息放在最容易看到的位置。
